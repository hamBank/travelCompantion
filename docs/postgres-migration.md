# SQLite → Postgres migration

Status: **Phases 1–3 done on production** (code dialect-agnostic + Alembic;
Postgres provisioned; all data copied and verified). The app **still runs on
SQLite** — Phase 4 (cutover) is pending.

> ⚠️ **Rotate the DB password at cutover.** The provisioning password was emitted
> in a migration log line before that was masked, so treat it as compromised.

> `provision_postgres.sh`) and use the new one in `DATABASE_URL`.

## Live-server bootstrap mechanism (how Phase 2/3 ran)

The SSH user (`travelcomp`) can't `sudo`, but the webhook deploy runs as root.
So privileged steps go through the deploy:
- `deploy.sh` installs Postgres if missing and, when a password is present in
  `$APP_DIR/.pg-bootstrap` (a `travelcomp`-writable sentinel) or
  `PG_BOOTSTRAP_PASSWORD` in `.env`, idempotently ensures the `travelcomp`
  role + database exist.
- Schema + data then run as `travelcomp` (no root needed):
  `alembic upgrade head` and `scripts/migrate_to_postgres.py` against the PG URL.

Verified: all 9 tables row-for-row equal to `travel.db` (513 rows); sequences
advanced past max ids.

## Why / what changed in Phase 1

- `DATABASE_URL` env var selects the backend; defaults to `sqlite:///./travel.db`.
  SQLite-only `connect_args` are applied only for sqlite URLs
  (`backend/database.py:_connect_args`).
- `data_version` (the cross-device sync signal in `/health`) no longer reads the
  `travel.db` file mtime — that doesn't exist on Postgres. It's now an in-process
  counter bumped by a SQLAlchemy `after_flush` listener, identical on both
  engines (`backend/database.py`). Strictly increases on every write.
- Schema is owned by **Alembic** on Postgres. `create_all()`/`_migrate()` now run
  only for sqlite (dev/test); on Postgres, `alembic upgrade head` builds the
  schema at deploy time. Data backfills still run on startup on both.
- A drift guard (`tests/test_alembic_drift.py`) runs the migrations onto a
  throwaway DB and asserts autogenerate finds no diff vs the models.

## Alembic day-to-day workflow

The models are the source of truth; every schema change needs a revision.

```bash
# after editing backend/models.py
alembic revision --autogenerate -m "describe the change"   # review the file!
alembic upgrade head                                        # apply locally (sqlite)
python -m pytest tests/test_alembic_drift.py               # must stay green
```

Notes:
- The URL is **not** in `alembic.ini`; `alembic/env.py` reads `DATABASE_URL`.
- Autogenerated revisions reference `sqlmodel.sql.sqltypes.*`; the
  `import sqlmodel` line is injected via `alembic/script.py.mako` — keep it.
- conftest uses `create_all()` for speed; the drift guard is what keeps the
  fast test schema and the migrations from diverging.

## How the baseline revision was created (for reference)

The baseline (`alembic/versions/*_baseline_schema.py`) represents the current
schema, generated by autogenerating against an **empty** DB:

```bash
DATABASE_URL="sqlite:///./_tmp.db" alembic revision --autogenerate -m "baseline schema"
rm _tmp.db
# add `import sqlmodel` to the generated file (now automatic via the template)
```

Verified clean: `alembic upgrade head` on an empty DB + `alembic check` →
"No new upgrade operations detected."

> The legacy `travel.db` reports cosmetic type diffs (`kind` VARCHAR vs Enum,
> `details` TEXT vs JSON) because it predates those model types. SQLite never
> enforced them. This is exactly why the cutover **builds a fresh Postgres schema
> and copies data** rather than converting in place — the legacy types never
> become Postgres DDL.

## Phase 2 — Provision Postgres (server)

```bash
sudo apt-get install -y postgresql
sudo -u postgres psql -c "CREATE ROLE travelcomp LOGIN PASSWORD '<pw>';"
sudo -u postgres psql -c "CREATE DATABASE travelcomp OWNER travelcomp;"
```

Add to `/opt/travelcomp/.env`:

```
DATABASE_URL=postgresql+psycopg://travelcomp:<pw>@localhost/travelcomp
```

`deploy.sh` additions (Phase 2 work, not yet applied):
- install `postgresql` package (first deploy only)
- run `alembic upgrade head` **before** restarting the service
- replace the implicit SQLite-file backup with `pg_dump` in the backup step
- add a CI job running the full pytest suite against a real Postgres (catches
  JSON/enum/datetime dialect issues the sqlite suite can't)

## Phase 3 — One-time data copy

`scripts/migrate_to_postgres.py` (to be written): open both engines, copy tables
in FK order (Trip → Stop → ItineraryItem → TripMembership → PendingChange →
IngestedEmail → ProcessedDocument → UserImportToken → ItemHistory) via SQLModel,
then reset Postgres sequences to `max(id)+1`. Dataset is small and single-user,
so a straight ORM copy is enough — no pgloader.

```bash
# fresh schema on Postgres first
DATABASE_URL="postgresql+psycopg://…/travelcomp" alembic upgrade head
# then copy
python scripts/migrate_to_postgres.py \
    --from sqlite:///./travel.db \
    --to postgresql+psycopg://…/travelcomp
```

Dry-run against a scratch DB; verify row counts per table and render a timeline.

## Phase 4 — Cutover

1. `pg_dump`/copy backup; stop the service.
2. `cp travel.db travel.db.bak` (rollback safety).
3. `alembic upgrade head` on Postgres, run the data-copy script.
4. Set `DATABASE_URL` to Postgres in `.env`, start the service.
5. Smoke test: two browsers, edit a cost on one, confirm the other syncs within
   30s (validates the new `data_version` counter on Postgres).
6. Keep `travel.db.bak` for one release before deleting.
